# 网络模型

## OSI 七层模型（理论标准）

| 层 | 作用 | 协议举例 |
|---|------|---------|
| **7. 应用层** | 用户接口、应用协议 | HTTP、FTP、DNS |
| **6. 表示层** | 数据格式转换、加密压缩 | SSL/TLS |
| **5. 会话层** | 建立/管理/终止会话、断点续传 | 合并到应用层 |
| **4. 传输层** | 端到端可靠传输 | TCP、UDP |
| **3. 网络层** | 路由寻址 | IP、ICMP |
| **2. 数据链路层** | 帧传输、差错控制 | Ethernet |
| **1. 物理层** | 比特流 | 网线、光纤 |

## TCP/IP 五层模型（实际用的）

| 层 | 职责 | 协议举例 |
|---|------|---------|
| **5. 应用层** | 为用户应用提供服务 | HTTP、DNS、WebSocket |
| **4. 传输层** | 端到端通信、可靠/不可靠 | TCP、UDP |
| **3. 网络层** | 路由寻址、分组转发 | IP、ICMP |
| **2. 数据链路层** | 相邻节点间帧传输 | Ethernet、PPP |
| **1. 物理层** | 比特流传输，电气信号 | 网线、光纤 |

## 区别

OSI 是理论模型，TCP/IP 是实际用的。OSI 把会话层、表示层、应用层拆成三层，TCP/IP 合并为一层应用层。平时开发主要接触的是 TCP/IP 五层模型。

# 输入URL到页面响应

用户输入 URL 回车 → DNS 解析拿到 IP → TCP 三次握手建立连接 → TLS 握手（HTTPS）协商会话密钥 → 发送 HTTP 请求 → 服务器处理并返回响应 → 浏览器渲染页面


1. **DNS 解析**：浏览器查本地 hosts → 查 DNS 缓存 → 递归查询 DNS 服务器，拿到目标 IP
2. **TCP 三次握手**：客户端与服务端建立可靠连接（SYN → SYN-ACK → ACK）
3. **TLS 握手**：如果是 HTTPS，四轮握手协商出对称会话密钥
4. **发送 HTTP 请求**：浏览器构造请求报文发送给服务器
5. **服务器处理**：反向代理（Nginx）→ Web 容器 → 业务逻辑 → 查询数据库 → 组装响应
6. **返回 HTTP 响应**：服务器返回 200 OK + HTML 内容
7. **浏览器渲染**：解析 HTML 构建 DOM 树 → 解析 CSS 构建 CSSOM 树 → 布局树 → 绘制，遇到图片/脚本资源继续发请求下载
# http/https

## https过程

```
TCP 三次握手后 → TLS 四轮握手
```

1. client发送clientHello到server,包括TLS版本、加密算法、client随机数
2. server接收clientHello,选定版本，发送ServerHello,包括server随机数、数字证书
3. client接收serverHello并用CA公钥验签数字证书,取出server公钥,加密pre-master随机数并发送给server
4. server用私钥解密得到pre-master,双方基于三个随机数生成对称加密会话密钥，建立https连接
建立连接后通过会话密钥传输信息
## https怎样防范中间人攻击
CA 证书体系：CA 不会随便给中间人签服务器的证书，中间人拿不出合法证书，客户端验签就失败，攻击暴露

## HTTP 与 HTTPS 区别

|         | HTTP           | HTTPS                      |
| ------- | -------------- | -------------------------- |
| **协议**  | 明文传输           | TLS 加密传输                   |
| **端口**  | 80             | 443                        |
| **安全性** | 无加密，可被窃听/篡改/冒充 | 加密防窃听 + 证书验证防冒充            |
| **证书**  | 不需要            | 需要 CA 签名的证书                |
| **性能**  | 快（无加密开销）       | 略慢（TLS 握手 + 加解密开销，但现在差异很小） |
| **SEO** | 浏览器标记"不安全"     | 浏览器标记"安全锁"，搜索引擎优先收录        |
