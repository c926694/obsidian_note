# 网络模型

## OSI 七层模型（理论标准）

| 层 | 作用 | 协议举例 |
|---|------|---------|
| **7. 应用层** | 用户接口、应用协议 | HTTP、FTP、DNS |
| **6. 表示层** | 数据格式转换、加密压缩 | SSL/TLS |
| **5. 会话层** | 建立/管理/终止会话、断点续传 | 合并到应用层 |
| **4. 传输层** | 端到端可靠传输 | TCP、UDP |
| **3. 网络层** | 路由寻址 | IP、ICMP |
| **2. 数据链路层** | 帧传输、差错控制 | Ethernet |
| **1. 物理层** | 比特流 | 网线、光纤 |

## TCP/IP 五层模型（实际用的）

| 层 | 职责 | 协议举例 |
|---|------|---------|
| **5. 应用层** | 为用户应用提供服务 | HTTP、DNS、WebSocket |
| **4. 传输层** | 端到端通信、可靠/不可靠 | TCP、UDP |
| **3. 网络层** | 路由寻址、分组转发 | IP、ICMP |
| **2. 数据链路层** | 相邻节点间帧传输 | Ethernet、PPP |
| **1. 物理层** | 比特流传输，电气信号 | 网线、光纤 |

## 区别

OSI 是理论模型，TCP/IP 是实际用的。OSI 把会话层、表示层、应用层拆成三层，TCP/IP 合并为一层应用层。平时开发主要接触的是 TCP/IP 五层模型。

# 输入URL到页面响应

用户输入 URL 回车 → DNS 解析拿到 IP → TCP 三次握手建立连接 → TLS 握手（HTTPS）协商会话密钥 → 发送 HTTP 请求 → 服务器处理并返回响应 → 浏览器渲染页面


1. **DNS 解析**：浏览器查本地 hosts → 查 DNS 缓存 → 递归查询 DNS 服务器，拿到目标 IP
2. **TCP 三次握手**：客户端与服务端建立可靠连接（SYN → SYN-ACK → ACK）
3. **TLS 握手**：如果是 HTTPS，四轮握手协商出对称会话密钥
4. **发送 HTTP 请求**：浏览器构造请求报文发送给服务器
5. **服务器处理**：反向代理（Nginx）→ Web 容器 → 业务逻辑 → 查询数据库 → 组装响应
6. **返回 HTTP 响应**：服务器返回 200 OK + HTML 内容
7. **浏览器渲染**：解析 HTML 构建 DOM 树 → 解析 CSS 构建 CSSOM 树 → 布局树 → 绘制，遇到图片/脚本资源继续发请求下载
# http/https

## https过程

```
TCP 三次握手后 → TLS 四轮握手
```

1. client发送clientHello到server,包括TLS版本、加密算法、client随机数
2. server接收clientHello,选定版本，发送ServerHello,包括server随机数、数字证书
3. client接收serverHello并用CA公钥验签数字证书,取出server公钥,加密pre-master随机数并发送给server
4. server用私钥解密得到pre-master,双方基于三个随机数生成对称加密会话密钥，建立https连接
建立连接后通过会话密钥传输信息
## https怎样防范中间人攻击
CA 证书体系：CA 不会随便给中间人签服务器的证书，中间人拿不出合法证书，客户端验签就失败，攻击暴露

## HTTP 与 HTTPS 区别

|         | HTTP           | HTTPS                      |
| ------- | -------------- | -------------------------- |
| **协议**  | 明文传输           | TLS 加密传输                   |
| **端口**  | 80             | 443                        |
| **安全性** | 无加密，可被窃听/篡改/冒充 | 加密防窃听 + 证书验证防冒充            |
| **证书**  | 不需要            | 需要 CA 签名的证书                |
| **性能**  | 快（无加密开销）       | 略慢（TLS 握手 + 加解密开销，但现在差异很小） |

## HTTP 版本对比

|           | HTTP/1.0       | HTTP/1.1                   | HTTP/2            | HTTP/3           |
| --------- | -------------- | -------------------------- | ----------------- | ---------------- |
| **连接**    | 短连接，每次请求新建 TCP | **长连接**（keep-alive），复用 TCP | 多路复用，一个 TCP 并发多请求 | 基于 **QUIC**（UDP） |
| **队头阻塞**  | 有              | 有                          | **解决了**           | 解决了              |
| **头部压缩**  | 无              | 无                          | **HPACK**         | **QPACK**        |
| **服务器推送** | 无              | 无                          | ✅                 | ✅                |

演进：1.0 用一次断一次 → 1.1 长连接省握手 → 2 多路复用 + 头压缩解决队头阻塞 → 3 换 UDP 连队头阻塞都解决了

### 队头阻塞

HTTP/1.1 一条 TCP 连接一次只能传一个请求/响应，前一个没完后面的就得等着。HTTP/2 把一个 TCP 切成多个流（stream），请求和响应可以交错发，互不阻塞。

### 头部压缩

HTTP 请求头每次都是重复的纯文本（Host、Cookie 等），浪费带宽。HTTP/2 用 HPACK 算法压缩，第一次完整传，后面只传变化的字段，引用之前的(通过维护索引表传索引号)。HTTP/3 用 QPACK，原理一样。

### 服务器推送

服务器返回 HTML 时，顺带把 HTML 里引用的 CSS、JS 等资源主动推给客户端，省得客户端再发请求去要。不过实际用得不多。

## HTTP 常见请求方法

| 方法 | 含义 | 请求体 | 典型场景 |
|------|------|--------|---------|
| **GET** | 获取资源 | ❌ | 查数据、打开页面 |
| **POST** | 提交/创建资源 | ✅ | 创建订单、登录 |
| **PUT** | 全量更新 | ✅ | 替换整个资源 |
| **DELETE** | 删除资源 | ❌ | 删一条记录 |
| **HEAD** | 只获取响应头 | ❌ | 检查资源是否存在、Content-Length |

- GET 参数在 URL 上，POST 参数在 body 里
- PUT 是全量替换（没传的字段会被清空），局部更新用 PATCH

## GET vs POST 区别

| 维度        | GET                                       | POST                                |
| --------- | ----------------------------------------- | ----------------------------------- |
| **参数位置**  | URL 上（?key=value）                         | Request body 里                      |
| **缓存**    | ✅ 可被浏览器缓存                                 | ❌ 不缓存                               |
| **长度限制**  | 有（URL 最长 2KB~8KB，取决于浏览器和服务端）              | 无（理论上不限）                            |
| **安全性**   | **不安全**，URL 明文暴露在地址栏、历史记录、服务端日志           | body 不暴露在 URL 上，但也是明文（HTTPS 才有真正安全） |
| **编码**    | URL 编码（application/x-www-form-urlencoded） | 多种：form-data、JSON 等                 |
