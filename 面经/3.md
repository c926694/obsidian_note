# 我看你第二个分布式小缓存里面写了一致性哈希，一致性哈希可以说一下吗？

一致性哈希的话，是针对多实例缓存节点的分配问题。如果有多台机器，每台机器管理一部分缓存，key 传过来给 value。普通哈希就是直接取模，对 key 进行哈希之后对机器数量取模，得到分配到哪台机器上。但这会导致如果增加机器，就会有数据失效、分布不均的情况。一致性哈希就是搞一个哈希环，对这个环哈希之后去确定机器的位置。

# 增加节点或删除节点会发生什么？

增加节点或删除节点，原本有一台机器上有缓存，增加节点之后使得查找的节点位置变了，打到另一台机器上，另一台机器没有缓存，需要重新建立缓存。而一致性哈希通过虚拟节点，使得数据分布更均匀。

# 唯一索引的话一般会怎么用的？

# 我有三个字段 A、B、C，我给 A、B 加联合索引，查询条件是 A、B、C，这个联合索引会生效吗？

会生效。联合索引遵循**最左前缀原则**：索引 (A, B) 会先按 A 排序，相同 A 内再按 B 排序。查询条件 `WHERE A=? AND B=? AND C=?` 中，A 和 B **走索引**，C 虽然不在索引的前缀列中，但可以通过**索引下推（ICP）**在索引层面做过滤，减少回表次数。所以三个字段都能利用到这个联合索引。

# 什么意思？查 ABC 实际上的查询执行流程什么样子？

具体流程：MySQL 从联合索引 (A, B) 的 B+ 树中，先按 A 定位到范围，再在 A 相同的数据中按 B 定位。定位到 A 和 B 匹配的索引行后，因为 C 不在索引前缀列，MySQL 不会跳过，而是利用**索引下推（ICP）**，直接在索引层面判断 C 条件是否满足，满足才回表读取完整行，不满足就跳过。这样可以大幅减少回表次数。所以 A、B 是最左前缀匹配，C 是通过 ICP 在索引中过滤。

# 可以用 UUID 去作为 MySQL 的主键吗？

用 UUID 的话，它是无序的吧？可以保证唯一性，但是它是无序的。主键索引是排序的，插入数据的时候因为是无序的要保证索引树是有序的，插入效率肯定不如递增的字段。还有 UUID 比较大，占的内存大一点。

# 一段程序计算从 1 加到 10，用数组存和用链表存循环累加，有什么区别吗？

数组和链表的区别？数组内存分布是连在一起的，可以直接通过偏移量寻址；链表通过指针进行寻址，这是最主要的区别。

# 性能呢？

数组直接寻址更快一点，直接通过偏移量就能找到下一个元素。

# 具体说一下它快到哪里？哪些地方它可能比较快？

主要在于**CPU 缓存局部性**：数组是连续内存，当 CPU 加载一个元素时，会把相邻的连续数据一起加载到 CPU 缓存行（cache line）中，后续遍历时大部分数据已在缓存中，命中率极高。而链表节点在内存中离散分布，每个节点都需要从主存加载，频繁**缓存未命中**，遍历效率差很多。偏移量寻址只是表层原因，缓存局部性是性能差距的根本。

# 说一下进程和线程的区别

进程是操作系统资源分配的单位，线程是调度的基本单位。一个是资源分配单位，一个是调度基本单位。线程是蕴含在进程里面的。进程切换需要保存的上下文比较多，因为比较大；线程切换成本相对较低。

# 这是哪里低啊？可以具体说一下吗？

线程切换比进程切换成本低，具体体现在两点：

1. **地址空间不变**：线程共用进程的地址空间，切换时**不需要切换页表**，也就无需刷新 TLB（快表）；进程切换必须切换页表，TLB 全部失效，重新加载开销大。
2. **上下文保存内容少**：线程切换只保存/恢复寄存器、栈指针、PC 等；进程切换还要保存/恢复内存映射、文件描述符表、信号处理等整套资源。

所以线程切换的开销比进程切换小一个数量级。

# 多进程和多线程有什么区别？

多进程和多线程的主要区别：

1. **内存隔离 vs 共享**：多进程地址空间互相隔离，一个进程崩溃不影响其他进程，稳定性高；多线程共享进程地址空间，一个线程非法操作会导致整个进程退出。
2. **通信方式**：多进程需要 IPC（管道、共享内存、消息队列等），较复杂；多线程通过共享内存可以直接读写，但需要锁或原子操作保证同步。
3. **创建/切换开销**：多进程创建和切换需要分配独立地址空间、页表等，成本高；多线程共用地址空间，创建和切换成本低。
4. **适用场景**：对稳定性要求高用多进程（如浏览器每个标签页一个进程）；对性能/响应速度要求高用多线程（如 Web 服务器处理请求）。

# 进程通讯方式有哪些？

进程间通讯（IPC）的常见方式有：

1. **管道（Pipe）**：包括匿名管道（父子进程间）和命名管道（FIFO，任意进程间）。
2. **消息队列（Message Queue）**：进程间通过消息队列传递结构化数据。
3. **共享内存（Shared Memory）**：一块内存区域映射到多个进程的地址空间，效率最高，但需要信号量同步。
4. **信号量（Semaphore）**：用于进程间同步，不传数据。
5. **信号（Signal）**：异步通知机制，如 SIGINT、SIGKILL。
6. **Socket**：可用于同一主机或不同主机的进程通信。

# 还是累加求和，从 1 加到 1 亿，一个线程算和两个线程分别算前半后半，运行在单核服务器上，有什么差别？

单核上多线程**反而更慢**，不是"没什么区别"。因为单核一次只能执行一个线程，多线程需要不断地进行**上下文切换**（保存/恢复寄存器、栈、PC 等），切换本身就有开销。而单线程没有切换开销，CPU 一直执行同一个线程，数据局部性也好，所以单核上**单线程比多线程更快**。

# 说一下 HTTPS 的工作原理

HTTPS 是 HTTP + TLS/SSL 的安全连接。核心流程如下：

1. **TCP 三次握手**后，客户端发送 ClientHello（支持的加密套件、TLS 版本等）。
2. **服务器**回复 ServerHello（选定加密套件），并将**数字证书**（包含服务器公钥，由 CA 事先签名）发给客户端。
3. **客户端验证证书**：用 CA 的公钥解密证书上的数字签名，对比证书内容的哈希值，如果一致则证书未被篡改。验证通过后，客户端生成一个随机的**对称密钥（会话密钥）**，用服务器的公钥加密后发给服务器。
4. **服务器**用自己的**私钥**解密，拿到对称密钥。
5. 之后的通信都使用这个对称密钥进行对称加密传输。

关键点：数字签名是由**CA（证书颁发机构）** 的私钥对证书内容签名生成的，不是服务器"通过中间人现场生成"。服务器只负责把 CA 签好的证书发给客户端，客户端用 CA 的公钥验证。

# 你刚才说中间人，那个中间人是干什么的？

中间人就相当于一个客观的存在，保证安全的可信第三方，有类似签名什么的。保证不被篡改，因为如果被篡改了客户端或服务器就能知道这是假的，就不会建立通信。数字签名能保证不被篡改。

# 怎么保证不被篡改呢？数字签名又怎么生成的？

数字签名的生成流程：CA 用哈希算法（如 SHA-256）对证书内容计算出一个**摘要**，再用 CA 自己的**私钥**对这个摘要进行加密，加密后的结果就是**数字签名**。客户端验证时，用 CA 的**公钥**解密数字签名得到摘要①，再自己对证书做同样的哈希运算得到摘要②，对比两个摘要是否一致。如果一致，说明证书内容没有被篡改，且确实是该 CA 签发的。
