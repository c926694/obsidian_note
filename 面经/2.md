## 数组和切片的区别

| 对比项 | 数组 | 切片 |
|--------|------|------|
| **长度** | 固定，且是类型的一部分（`[5]int` 和 `[3]int` 是不同类型） | 可变，长度不是类型的一部分 |
| **传递方式** | **值传递**（传参会拷贝整个数组，大数组开销大） | **引用传递**（传参只拷贝 slice 结构体，底层数组共享） |
| **内存** | 直接在栈/静态区分配连续内存 | 结构体（ptr + len + cap）指向堆上的底层数组 |
| **初始化** | `var a [5]int` 直接可用 | 需要 `make` 或字面量初始化，否则为 nil |
| **扩容** | 不支持 | 支持，自动扩容（1.18 前双倍，之后按 256 边界调整） |

## make和new区别

new针对任意类型，make只能用于slice、map、channel

new(T) 分配内存并**零值初始化**，返回 *T（指针）。比如 new([]int) 返回 *[]int，指向一个 nil 切片，所以无法使用 s[0]。

make 只用于 slice、map、channel，它会初始化底层数据结构并设置好内部字段（如 slice 的指针、长度、容量），返回**对象本身**（非指针）。比如 make([]int, 5) 得到一个长度为 5、元素都为 0 的切片，可以正常访问 s[0]。

总结：new 返回指针，零值初始化；make 返回对象，内部数据结构已就绪。

## 怎样实现有序map

Go 原生 map 是无序的，两种实现有序遍历的思路：

1. **手动排序**：把 key 取出放入切片 → 排序切片 → 按序遍历取值
   ```go
   keys := make([]string, 0, len(m))
   for k := range m { keys = append(keys, k) }
   sort.Strings(keys)
   for _, k := range keys { fmt.Println(k, m[k]) }
   ```
   适合数据量不大、偶尔需要有序的场景。

2. **第三方有序 map**：使用 `github.com/elliotchance/orderedmap`，内部用链表维护插入顺序，或者自己用 B+ 树 / 跳表实现。适合频繁需要有序访问的场景。

## map是并发安全的吗

不是普通 map 不是并发安全的。sync.Map 是并发安全的，它的实现不是简单的互斥锁，而是采用**读写分离 + 原子操作**：读操作优先从 read 映射中无锁读取（用 atomic.Load），写操作通过 miss 次数的判断决定是否需要将 read 映射提升，只有涉及 dirty 映射操作时才加互斥锁。适合读多写少的场景。

## 控制并发方式

多种方式：

1. **WaitGroup**：等待一组 goroutine 完成
2. **Channel**：通过 channel 做信号通知或协程间同步
3. **Context**：通过 cancel / timeout 控制协程生命周期
4. **互斥锁 / 读写锁**：控制共享资源并发访问
5. **Worker Pool**：通过带缓冲 channel 限制同时运行的 goroutine 数量，避免无限制创建

## 索引分类

按不同维度分类：

1. **数据结构**：B+树索引（MySQL 默认）、Hash 索引、全文索引
2. **物理存储**：聚簇索引（数据和索引在一起）、二级索引（索引存主键引用）
3. **字段数**：单列索引、联合索引（组合索引）
4. **业务逻辑**：主键索引、唯一索引、普通索引

## 索引失效场景

1. **LIKE 前模糊匹配**：`LIKE '%abc'` 不走索引，`LIKE 'abc%'` 走
2. **OR 两边字段不全是索引列**：OR 任意一侧没索引就可能全表扫描
3. **对索引列使用函数**：如 `WHERE YEAR(create_time) = 2024`（不是"聚合函数"，是对列用了函数）
4. **隐式类型转换**：如字段是 varchar，传入数字 `WHERE name = 123`
5. **不等号条件**：`<>, !=` 通常不走索引
6. **复合索引不满足最左前缀**：索引 (A,B)，只查 B 不走索引

## redis实现分布式锁

### 加锁

```go
SET lock_key unique_id NX PX 3000
```
- `NX`：不存在时才能设置（互斥）
- `PX 3000`：锁自动过期 3s（防止死锁）
- `unique_id`：当前请求的唯一标识（如 UUID），用于安全释放锁

### 安全释放锁

释放时不能直接 DEL，要先用 unique_id 判断锁是否还是自己的，避免误删别人的锁：

```lua
-- Lua 脚本保证原子性
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

### 面试官追问：锁过期时间 3s，业务执行 4s 怎么办

这里其实涉及**两个不同的问题**，都要答清楚：

**问题一：锁被自动释放，别的请求拿到锁，当前线程还在执行 → 出现并发**
→ 解决：**看门狗（Watchdog）机制**。后台起一个守护协程，定期（如过期时间的 1/3）检查锁还在不在、业务是否还在执行，如果是就续期（重置过期时间）。Redisson 的 Watchdog 就是干这个的。

**问题二：当前线程执行完要释放锁，但锁可能已经过期被其他线程占用了 → 误删别人的锁**
→ 解决：value 存**唯一 ID**（如 UUID + 线程 ID），释放时用 Lua 脚本先比对再删除。这样即使锁过期被别的线程占了，也不会删错。

### 最佳实践总结

| 问题 | 解决方案 |
|------|----------|
| 防止死锁 | 设置过期时间 |
| 误删别人的锁 | value 存唯一 ID + Lua 脚本释放 |
| 业务执行超时 | 看门狗自动续期 |
| 加锁失败 | 自旋重试 + 随机退避 |
